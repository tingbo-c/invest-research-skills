# Industry Research Skill — 输出模式设计评估报告

## 总判断

整体思路正确，框架骨架已经是 B+ 水平，但有一个结构性缺口未补上。

**值得保留：**
- 三档分流按"使用场景 + 严谨度"切割，逻辑清晰
- 按需加载 reference 的设计克制，有效控制 token 和主文件复杂度
- 平台无关性做得好，README 的三种用法对应不同受众
- `output-structure.md` 的 SCQR 原则有实用价值

**核心问题：当前三档分流混淆了两个正交维度：严谨度（quick/standard/disclosure）和长度/生成方式（short/standard/long）。**

长报告不是"更严谨的标准报告"，而是一种不同的生成工作流。不把这个区分出来，遇到"几万字白皮书"需求时，skill 没有明确指引。

---

## 一、输出模式建议

### 现有三档实质是"严谨度轴"

| 模式 | 实质 |
|---|---|
| 快速判断版 | 低严谨度、低篇幅 |
| 标准报告版 | 中严谨度、中篇幅 |
| 披露/底稿版 | 高严谨度、任意篇幅 |

问题：`披露/底稿版` 是关于来源可追溯性的，不是篇幅深度的。"几万字深度报告"无法被现有三档识别。

### 建议：显式加入"长报告工作流"

长报告模式不是第四档深度，而是一种**生成工作流标识符**。

**触发词：**
- "几万字" / "深度报告" / "白皮书" / "完整版研究报告"
- 篇幅暗示词："分章节" / "可以慢慢写" / "不用一次输出"

**SKILL.md 输出模式部分修改为：**

```markdown
| 用户意图 | 默认输出 |
|---|---|
| "分析一下""研究一下""值不值得看" | 快速判断版 |
| "完整分析""详细报告""按结构输出" | 标准报告版 |
| "招股书""募资材料""可追溯" | 披露/底稿版 |
| "几万字""深度报告""白皮书""不用一次输出" | 长报告工作流 |
```

长报告模式单独作为一节放在 SKILL.md 输出模式之后，明确声明"这是一个多轮工作流，不是一次输出"。

### 建议：把"披露/底稿版"重定义为严谨度修饰符

披露严谨度可以叠加到任意输出模式上，而不是作为独立的第三档：

- 快速判断版 + 披露严谨度 = 招股书摘要
- 标准报告版 + 披露严谨度 = 完整底稿
- 长报告工作流 + 披露严谨度 = 完整可追溯白皮书

## 二、模板设计建议

### 当前问题

`references/output-structure.md` 只有 9 个章节标题 + SCQR 原则，这是"目录"，不是"模板"。目录无法防止 agent 输出空洞内容，也无法让人类分析师知道每章应该写到什么程度。

### 推荐方案：半结构模板，放在 `assets/report-template.md`

每个章节一组，包含以下字段：

```markdown
## 章节名称

**最小写作要求**
- 字数范围提示（如"200-400字"）
- 必须包含的子项（如"必须有产业链结构描述"）

**必须回答的问题**
- 问题1（如"行业边界按哪个分类标准？为什么？"）
- 问题2
- 问题3

**不能犯的错误**
- 错误1（如"不能只给行业名称，不说边界在哪里截止"）
- 错误2

**需要的数据/来源类型**
- 优先来源（如"申万/证监会分类文件"）
- 最低要求（如"至少说清楚上下游截止点"）
```

### 不应该放进模板的内容

- 示例文本或样例段落（过拟合特定行业）
- 具体数字标准（如"市场规模必须大于 500 亿"）
- 格式化规则（字体、颜色、图表样式——平台相关）
- 跨章引用逻辑（应由 agent 自行推导）
- 行业特定的估值倍数（已有 `references/valuation-multiples.md` 负责）

### 职责划分调整

| 文件 | 应该放什么 |
|---|---|
| `references/output-structure.md` | 章节顺序、SCQR 汇报原则、三档深度定义 |
| `assets/report-template.md` | 每章的最小要求 / 必答问题 / 常见错误 |
| `SKILL.md` | 何时加载模板（标准报告版和长报告工作流时加载） |

## 三、长报告工作流建议

### 为什么必须拆章生成

1. 一次性生成容易在后半段"遗忘"前半段的逻辑假设
2. 章节之间的数据引用、逻辑推导需要在展开时显式检查
3. 用户有机会在提纲阶段纠偏，避免写完 2 万字再推倒重来

### 工作流定义（三段式）

建议在 SKILL.md 新增一节，约 15-20 行骨架：

```markdown
## 长报告工作流

触发条件：用户意图含"几万字"/"深度报告"/"白皮书"/"分章"/篇幅明确超出标准报告时。

### 第一段：提纲确认
- 根据行业特点和用户要求，生成完整章节提纲
- 每章标注预估字数范围、核心回答问题
- 提纲确认后再开始写作（如有用户交互，等确认；如无，说明后继续）

### 第二段：分章展开
- 每次只展开 1-2 章
- 每章写完时主动交代：本章完成、下一章是什么
- 涉及上一章数据时，明确引用（"按第三章测算的 TAM..."）
- 加载 `assets/report-template.md` 获取每章写作要求

### 第三段：结构复核
- 全部章节完成后，做跨章一致性检查
- 检查项：数据口径是否一致、结论是否前后矛盾、关键数字来源是否统一
- 加载 `references/analysis-review-checklist.md`
```

### 主文件应该写到什么程度

SKILL.md 只需要：
1. 识别触发词 → 声明进入长报告工作流
2. 三段式流程的骨架（3-5 行每段）
3. 加载指令（指向 `assets/report-template.md` 和 `references/analysis-review-checklist.md`）

具体"每章写什么、最小要求"全部放 `assets/report-template.md`，不放主文件。

## 四、结构职责建议

### 各文件最合理的职责边界

| 文件/目录 | 职责 | 不应该放的 |
|---|---|---|
| `SKILL.md` | 地图：场景分流、输出模式识别、加载规则、工作流骨架 | 具体写作要求、公式细节、来源列表 |
| `README.md` | 使用手册：三类用户用法、目录说明、任务→文件映射 | 方法论细节 |
| `references/` | 方法论库：公式、数据质量、估值、时间口径、复核 checklist | 写作模板、输出格式 |
| `assets/` | 输出资源：报告模板（每章要求）、案例片段 | 方法论、数据规则 |
| `scripts/` | 可执行计算工具 | 分析判断逻辑 |

### "如何写成专业报告"的分配

- "应该写哪些内容/章节" → `references/output-structure.md`（已有）
- "每章应该写到什么程度/避免什么" → `assets/report-template.md`（待建）
- "长篇报告如何组织生成过程" → `SKILL.md` 新增一节

### 关于扩展后会不会变重

只要坚持"主文件是地图，不是百科全书"的原则，就不会变重：
- SKILL.md 新增长报告工作流时，只写触发条件 + 三段骨架 + 加载指令，不超过 20 行
- 具体模板内容移到 `assets/`，按需加载
- 不要把"第一章应该写什么"这类内容塞进 SKILL.md

## 五、最佳实践

### 推荐的输出策略：意图识别 + 工作流路由

```
用户输入 → 识别意图：
  1. 简洁分析  → 快速判断版（单次输出，不加载模板）
  2. 完整结构  → 标准报告版（单次输出，加载模板）
  3. 长篇深度  → 长报告工作流（多轮，分阶段）
  + 任意档位叠加 → 披露严谨度（disclosure-grade checklist）
```

### 800字摘要到3万字长报告的产品化方式

核心思路：让**用户意图信号**驱动工作流选择，而不是让 agent 猜测应该写多少。

1. 默认快速判断版（除非有明确"完整/长篇"信号）
2. 触发标准报告版：明确要求结构/完整/报告格式
3. 触发长报告工作流：明确要求几万字/白皮书/不限长度/分章
4. 叠加披露严谨度：招股书/募资/公开发布关键词

这套规则可以被任何 agent 理解，不依赖 Claude 特有的 runtime 能力。

## 结论：当前方案是否已经足够？

**不足够，还差两件事才算完整：**

### 必须补

1. **`assets/report-template.md`（半结构模板）**
   当前 `output-structure.md` 只有目录，无法锚定每章质量。标准报告版和长报告工作流都需要这个。

2. **SKILL.md 的长报告工作流节**
   识别触发词 + 三段式流程（提纲→分章→复核）。不需要单独新建 reference，主文件里加一节（约 15-20 行骨架）即可。

### 可选但建议

3. **把"披露/底稿版"重定义为严谨度修饰符**
   允许它与快速判断版/标准报告版/长报告叠加，而不是作为独立的第三档，语义会更准确。
